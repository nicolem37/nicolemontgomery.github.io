Case Study

Agile E-Commerce QA Project

A full Software Testing Life Cycle engagement inside a simulated Agile Scrum team, testing a SaaS e-commerce application from requirements through regression.

01 — Project Overview

I served as QA Engineer inside a cross-functional Scrum team, contributing across the full STLC — from requirements analysis through defect verification — for a SaaS e-commerce web application.

Role
QA Engineer, Scrum Team
Scope
Full STLC — requirements through regression
Application
SaaS e-commerce platform
Tools
Jira, Zephyr, Postman, MySQL
02 — Business Challenge

A customer journey that had to hold up under real usage

The team needed confidence that a full customer journey — account creation, product search, cart, checkout, and payment — would perform reliably before release. Requirements, user stories, and design mockups didn't always agree, and gaps in that alignment carry real cost: abandoned carts, support tickets, lost revenue.

03 — My Testing Strategy

Start with requirements, not test cases

I reviewed Business Requirement Documents, user stories, and acceptance criteria first, to surface ambiguity before it became a defect. From there I designed manual test coverage across functional, regression, exploratory, smoke, and end-to-end scenarios, validated the API and database layers independently, and maintained a traceability matrix so every requirement had a test — and every test had a result.

04 — Functional Areas Tested
  • Account creation & login
  • Product search & filtering
  • Shopping cart
  • Checkout & discount codes
  • Payment processing
  • Order confirmation & history
  • Mobile & cross-browser responsiveness
05 — Defects Discovered

A valid discount code, rejected at checkout

During checkout testing, a promotional discount code that met every stated condition was rejected by the system — a defect that would have quietly reduced conversion and increased support volume if it reached production. I documented the full reproduction path, expected versus actual behavior, severity, priority, and business impact, then verified the resolution through regression testing before it was marked closed.

06 — QA Deliverables
  • Test plan & manual test cases
  • Requirement clarification questions
  • Requirement Traceability Matrix
  • Test execution results
  • Defect reports
  • API test collection
  • SQL validation queries
07 — Lessons Learned

This project reinforced that quality assurance extends far beyond executing test cases. Analyzing requirements before testing begins, asking clarifying questions when business rules are ambiguous, maintaining traceability between requirements and coverage, and validating both frontend and backend behavior — together, these are what let a team ship software people can actually rely on.

08 — Business Impact

This was a training engagement, not a live production release — but the discipline mirrored real stakes. The discount-code defect is a common, costly failure mode in e-commerce: a checkout bug that quietly increases cart abandonment and support volume. Catching it before sign-off, with a fully reproducible report and a verified fix, is exactly the kind of check that protects revenue and customer trust in a live environment.
